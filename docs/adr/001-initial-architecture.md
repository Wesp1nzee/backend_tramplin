# ADR 001: Initial Architecture

**Статус:** Принято
**Дата:** 2026-03-21
**Автор:** Backend Team

---

## Контекст

Разработка нового backend для платформы «Трамплин» — экосистемы для студентов и работодателей.

**Технологический стек:**
- FastAPI (веб-фреймворк)
- SQLAlchemy + Alembic (ORM, миграции)
- PostgreSQL + PostGIS (гео-запросы для вакансий)
- uv (менеджер пакетов и виртуальных окружений)

---

## Решение

Выбрана **слоистая архитектура** (Layered Architecture) со следующей структурой:

```
src/
├── api/                 # HTTP слой (endpoints, dependency injection)
│   └── v1/
│       ├── deps.py      # Зависимости для DI
│       └── endpoints/   # Роутеры по доменам
├── core/                # Глобальные настройки
│   ├── config.py        # pydantic-settings
│   ├── security.py      # JWT, hashing
│   └── exceptions.py    # Базовые исключения
├── db/                  # Инфраструктура БД
│   ├── session.py       # SessionManager, get_db
│   └── base.py          # DeclarativeBase, mixins
├── models/              # SQLAlchemy ORM модели
├── schemas/             # Pydantic DTO (Request/Response)
├── repositories/        # Слой доступа к данным
│   ├── base.py          # GenericRepository[T]
│   └── user.py          # UserRepository
└── services/            # Бизнес-логика
    └── auth.py          # AuthService
```

**Ключевые принципы:**

1. **Однонаправленные зависимости:**
   ```
   API → Services → Repositories → Models
   ```

2. **Repository Pattern:** Инкапсуляция SQL-запросов в репозиториях. Сервисы не работают с БД напрямую.

3. **Dependency Injection:** Все зависимости объявлены централизованно в `deps.py` и передаются через `Depends()`.

4. **DTO через Pydantic:** Схемы запросов/ответов отделены от ORM-моделей.

5. **Глобальная обработка ошибок:** Базовый класс `AppError` с автоматической конвертацией в HTTP-ответы.

## Рассмотренные альтернативы

### 1. Modular Monolith

**Описание:** Разбиение на модули по доменам (auth, users, vacancies, companies) с явными границами.

**Преимущества:**
- Лучшая изоляция доменов
- Возможность выделения модулей в отдельные сервисы

**Недостатки:**
- Избыточная сложность для MVP
- Требует больше времени на онбординг
- Сложнее кросс-доменные взаимодействия (например, auth + vacancies)

**Решение:** Отклонено. Вернёмся к этой архитектуре при масштабировании команды или выделении микросервисов.

---

### 2. Clean / Hexagonal Architecture

**Описание:** Архитектура с портами и адаптерами, где domain layer не зависит от инфраструктуры.

**Преимущества:**
- Максимальная тестируемость
- Независимость от фреймворков

**Недостатки:**
- Значительное увеличение количества файлов (interfaces, ports, adapters, injectors)
- Сложность для разработчиков, не знакомых с паттерном
- Over-engineering для текущего масштаба проекта

**Решение:** Отклонено. Может быть рассмотрено при переходе на микросервисы.

---

### 3. Django-like Structure (models + views + serializers)

**Описание:** Структура по типу Django (models, views, serializers в отдельных папках).

**Преимущества:**
- Знакомо разработчикам с Django-бэкграундом
- Меньше файлов за счёт интеграции ORM

**Недостатки:**
- FastAPI не предоставляет встроенной ORM
- Смешение бизнес-логики с HTTP-слоем в views
- Сложнее тестировать бизнес-логику изолированно

**Решение:** Отклонено. Предпочтено явное разделение через services/repositories.

## Последствия

### Положительные

| Аспект | Преимущество |
|--------|--------------|
| **Онбординг** | Новые разработчики быстро понимают структуру (стандартные слои) |
| **Тестирование** | Сервисы тестируются с моками репозиториев, без БД |
| **Изоляция** | Изменение SQL-запросов не затрагивает сервисы |
| **DI** | Легко подменять зависимости в тестах (test overrides) |
| **Масштабирование** | Можно добавлять слои (например, CQRS) без рефакторинга |

### Отрицательные

| Аспект | Недостаток |
|--------|------------|
| **Бойлерплейт** | Больше файлов: endpoint → service → repository → model → schema |
| **Навигация** | Приходится переключаться между папками для понимания потока |
| **Дублирование** | Схемы (schemas) частично дублируют модели (models) |

### Митигация недостатков

1. **Бойлерплейт:** Использование generic `BaseRepository[T]` сокращает код репозиториев.
2. **Навигация:** Документирование в `ARCHITECTURE.md` с диаграммами.
3. **Дублирование:** Pydantic v2 `from_attributes=True` уменьшает количество явных мапперов.
